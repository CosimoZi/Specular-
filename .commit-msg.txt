task #34 was a phantom: weapon id 13509 != CalamityQueller (13507)

The shenhe.test panel-baseline hardcoded weaponId 13509 "// Calamity
Queller" but the actual mapping in go-id-map.json is:
  13507 -> CalamityQueller
  13509 -> EngulfingLightning

So the engine was correctly computing Shenhe + EngulfingLightning
(base.atk = 911.83 because EL has substat enerRech_, not atk_, and
its main-stat curve is ATTACK_301 base 45.9364) while the test
asserted those numbers against expected CalamityQueller values. Looked
like an aggregation hole, was actually a typo.

Verified with both IDs in a new regression test
(weapon-aggregation.test.ts):

  Shenhe L90 A6 + EngulfingLightning L90 A6 R1:
    base.atk    = 911.83    (303.76 char + 421.37 weap_curve + 186.7 weap_asc)
    premod.atk_ = 0.288     (Shenhe A6 ascension only)
    enerRech_   = 0.551     (EL substat 0.12 * CRITICAL_301[90])
    final.atk   = 1174.44

  Shenhe L90 A6 + CalamityQueller L90 A6 R1:
    base.atk    = 1044.34   (303.76 char + 553.88 weap_curve + 186.7 weap_asc)
    premod.atk_ = 0.453     (Shenhe A6 0.288 + CQ substat 0.036 * CRITICAL_301[90])
    final.atk   = 1517.82

Both match wiki + raw stat data exactly. agg.reread(weaponSheet) works
correctly; charData and weaponData are structurally identical for
this purpose.

Changes:
  - shenhe.test.ts: fix weaponId 13509 -> 13507, update expected
    base.atk and premod.atk_ to true CalamityQueller values
  - data/index.ts: remove stale audit comment describing the phantom
    aggregation hole
  - new test/weapon-aggregation.test.ts: lock in the correct numbers
    for BOTH weapons so the typo class of bug is caught next time

Unblocks the broader weapon implementation plan.
